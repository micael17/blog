## AI 에이전트에게도 Git이 필요하다 — re_gent

### TL;DR
- Claude Code 같은 AI 코딩 에이전트가 코드를 수정할 때, 어떤 프롬프트가 어느 줄을 바꿨는지 추적해주는 버전 관리 도구다.
- `git blame`처럼 파일 각 줄이 "어떤 AI 대화에서 작성됐는지" 소급해 확인할 수 있다.
- Git을 대체하지 않고 보완한다. 두 도구를 나란히 쓰는 것이 목적이다.

---

### 이게 뭔가

Git은 개발자가 직접 커밋할 때 이력을 남긴다. 그런데 AI 에이전트가 세션 하나에서 파일을 수십 번 수정할 경우, Git 커밋 사이에는 아무 기록도 남지 않는다. "5분 전까지는 동작했는데"라며 에이전트에게 되돌리기를 요청해봤지만 대화 맥락이 끊겨 포기한 경험이 있다면, 정확히 그 공백을 채우는 도구가 re_gent다.

re_gent는 Claude Code와 훅(hook)으로 연동해 에이전트의 모든 툴 호출(파일 편집, 터미널 명령, 파일 생성)을 자동으로 **Step**이라는 단위로 기록한다. 각 Step에는 어떤 프롬프트가 변경을 유발했는지, 어떤 툴을 사용했는지, 해당 시점의 대화 내용까지 함께 저장된다. `.git/`처럼 `.regent/` 디렉토리에 데이터가 쌓이고, SQLite 기반 인덱스 덕분에 10ms 미만의 빠른 조회가 가능하다.

---

### 왜 지금 화제인가

2026년 4월 30일 공개 후 열흘도 되지 않아 GitHub 별 232개, 포크 12개를 기록했다. Hacker News "Show HN" 게시물이 109점, 댓글 47개를 받았는데, 핵심 문장 하나가 개발자들의 공감을 샀다. "우리는 에이전트에게 코드베이스 쓰기 권한을 줬다. 그런데 그에 대한 Git은 갖지 못했다." Claude Code, Cursor Agent 등 AI 코딩 에이전트 사용이 일상화되면서 에이전트 활동 자체를 감사(audit)하려는 수요가 급증하는 시점에 등장한 것이 화제의 이유다.

---

### 누가 써야 하나

- **AI 에이전트를 실무에 쓰는 개발자**: 세션 도중 "방금 뭘 바꿨지?"라는 의문이 든다면 바로 이 상황을 위한 도구다. `rgt log`로 세션 전체 변경 이력을, `rgt blame`으로 특정 줄의 출처 프롬프트를 추적할 수 있다.
- **팀 단위로 AI를 도입한 테크리드**: 에이전트가 어떤 맥락에서 어떤 코드를 작성했는지 감사 로그가 필요할 때. 코드 리뷰에서 "이 코드가 어떤 지시에서 나왔나요?"에 답을 줄 수 있다.
- **장시간 에이전트 세션을 운용하는 개발자**: `/compact`나 `/clear` 이후에도 대화 이력이 유실되면 안 되는 경우. re_gent는 세션 재시작과 무관하게 Step 기록을 유지한다.

### 빠르게 써보기

```bash
# Homebrew로 설치 (macOS/Linux)
brew tap regent-vcs/tap
brew install regent

# 또는 Go로 직접 설치
go install github.com/regent-vcs/regent/cmd/rgt@latest
```

```bash
# 프로젝트 디렉토리에서 초기화
cd your-project
rgt init

# Claude Code로 평소처럼 작업 — Step은 자동으로 기록됨

# 세션 전체 변경 이력 확인
rgt log

# 특정 줄이 어떤 프롬프트에서 왔는지 확인
rgt blame src/handler.go:42

# Step 하나의 상세 내용 (프롬프트 + 변경사항 + 대화) 확인
rgt show a1b2c3d

# 활성 세션 목록
rgt sessions
```

`rgt init` 후 Claude Code를 켜고 파일 몇 개를 수정하면 `rgt log`에서 툴 호출 단위의 이력이 즉시 보인다. 설치부터 첫 확인까지 5분이면 충분하다.

Step들이 DAG(유향 비순환 그래프) 구조를 이루고, 각 세션이 독립 브랜치를 갖습니다. 덕분에 에이전트 여러 개가 동시에 동작해도 충돌이 없습니다. Git의 브랜치 모델을 에이전트 동시성 문제에 적용한 발상입니다.

### 비슷한 도구와 비교

**Git과의 관계**: re_gent는 Git을 대체하지 않는다. Git이 개발자의 의도적 커밋을 관리한다면, re_gent는 에이전트의 자동 변경을 추적한다. 두 도구는 서로 다른 레이어에서 동작하므로 병행 사용이 기본이다. 아래 표가 핵심 차이를 정리한다.

| | Git | re_gent |
|---|---|---|
| 코드 변경 추적 | ✅ | ✅ |
| 에이전트 활동 추적 | ❌ | ✅ |
| 어떤 프롬프트가 이 줄을 썼나 | ❌ | ✅ |
| 대화 내용 보존 | ❌ | ✅ |
| 동시 세션 처리 | 충돌 위험 | 세션별 독립 브랜치 |

**단순 스냅샷 스크립트와의 차이**: 로컬 히스토리 기능이나 수동 스냅샷으로 비슷한 효과를 내려는 시도가 있지만, re_gent의 차별점은 "어떤 프롬프트가 이 변경을 유발했는가"라는 인과 관계를 추적한다는 데 있다. BLAKE3 기반 콘텐츠 주소 저장으로 중복 제거도 자동이다.

### 한 줄 평

Claude Code를 실무에 쓰고 있다면 지금 바로 써볼 가치가 있다. `rgt rewind` 같은 핵심 기능이 아직 미완성이라 프로덕션 의존은 이르지만, "에이전트 활동 감사 로그"라는 문제 자체는 실재하고 접근 방식은 올바르다. 공개 열흘 만에 HN에서 109점을 받은 것이 그 증거다.
